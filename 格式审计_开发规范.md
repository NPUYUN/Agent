# 格式审计 开发规范
本规范面向“格式审计智能体（FMT / Standardization_Auditor_Agent）”，以《数据库存入规则.md》的规则入库口径为核心，融合《开发规范.md》的系统级约束，定义格式审计在**规则建模/入库、数据库交互、API协议、业务闭环**上的统一要求。

## 1. 适用范围与目标
1. 本规范仅覆盖格式审计智能体（FMT）的开发与交付约束，不涉及计划中枢与其他专项智能体的实现细则。
2. 所有对外API输入输出必须严格符合JSON Schema与约定协议；不符合即视为接入失败。
3. 规则、论文切片、专家评语、任务结果均以数据库为事实源（source of truth），本地不做持久化依赖。

## 2. 通用技术环境规范（FMT必须遵守）
1. 编程语言：Python 3.10.x
2. Web框架：FastAPI（必须支持async异步处理，禁止阻塞式开发）
3. 数据模型：Pydantic V2（强制实现入参/出参类型校验）
4. 并发请求：使用httpx / aiohttp，严禁使用阻塞型requests库
5. CPU密集型/阻塞型任务：必须通过线程池或等价方式从事件循环中卸载，避免阻塞FastAPI事件循环

## 3. 系统架构与数据流向（FMT在全链路中的位置）
系统采用分布式多智能体架构并遵循四层闭环数据流向：
1. 数据摄入层：用户上传PDF → 中枢组将PDF转化为Markdown
2. 结构化层：中枢组将Markdown拆解为论文片段并入库（paper_sections）
3. 分布式审计层：中枢组通过异步API唤醒FMT → FMT读取论文片段与专家评语 → 生成JSON审计结果并实时写入review_tasks，同时将审计明细按统一格式写入agent_audit_result.result_json
4. 反思汇总层：反思评估组汇总各Agent结果形成最终评审报告

## 4. 规则入库与规则模型（核心）
### 4.0 系统规则主数据（main_rules / rule_judge）
1. 系统侧存在两张规则表：
   - `main_rules`：规则“元数据/基本信息”主表，记录系统当前有哪些规则项（主表：一）。
   - `rule_judge`（或等价详情表）：规则详情/判定口径（详情表：多），用于承载可扩展字段与不同规则类型的细化信息。
2. 智能体执行时应以`main_rules`中与自身`agent_code`匹配的规则为权威来源，输出结果中的`rule_id/score/level`需与系统规则口径一致，便于跨组汇总与统计。
3. 本仓库内的`agent_rules`与`rules.yaml`用于运行时参数驱动与兜底，不替代系统侧`main_rules`的指标主数据地位。

### 4.1 规则存储总原则
1. 规则以“可入库、可追溯、可热更新”为目标；规则的数据库存储口径以本章为准。
2. 规则分为两类：
   - 指标规则（评分/达标类）：用于描述FMT维度的评分项、阈值与严重性（例如FMT-001～FMT-004）。
   - 执行规则（运行时配置类）：用于驱动FMT的具体检测策略（例如排版/引用/标点/术语等模块参数）。
3. 在数据库中统一通过`agent_rules`表承载规则：每条规则一行，`rule_id`唯一标识，`content`为YAML/JSON字符串。

### 4.2 `agent_rules`表（规则承载表）
|字段名|类型|约束|说明|
|---|---|---|---|
|id|BigSerial|Primary Key|自增主键|
|rule_id|String|Unique, Indexed|规则唯一标识（例如FMT-001或typo_check等）|
|content|Text|Not Null|规则内容（YAML/JSON字符串）|
|updated_at|Timestamp|-|规则更新时间|

### 4.3 指标规则：通用字段模型（content内部结构）
指标规则的`content`需按以下通用字段组织（字段名固定，便于跨组统一统计与解释）：

|英文字段名|字段说明|字段类型|
|:---|:---|:---|
|rule_id|规则唯一标识|字符串|
|rule_name_en|规则名称（英文）|字符串|
|rule_name_cn|规则名称（中文）|字符串|
|rule_detail|规则详细描述（中文）|文本|
|full_score|单条规则满分值|数字（正整数）|
|severity|违规严重等级|枚举（CRITICAL/WARNING）|
|check_indicator|核查指标名|字符串|
|operator|比较运算符|枚举（\>/\</\>=/\<=/==）|
|threshold_val|判定阈值|字符串/数字|
|threshold_unit_en|阈值单位（英文）|字符串|
|threshold_unit_cn|阈值单位（中文）|字符串|
|agent_code|归属智能体编码|枚举（FMT/REF/EXP/LOG）|
|agent_name_en|归属智能体名称（英文）|字符串|
|agent_name_cn|归属智能体名称（中文）|字符串|

约束：
1. `agent_rules.rule_id`必须与`content.rule_id`一致。
2. `severity`仅允许`CRITICAL`或`WARNING`。
3. `full_score`为正整数，且FMT维度总分需满足本规范第4.4章约束。

### 4.4 FMT 指标规则清单（满分 20 分）
该清单为FMT维度的指标规则权威版本。每条指标规则可作为一条`agent_rules`记录入库（`rule_id`=FMT-XXX，`content`包含本节字段）。

|:---|:---|:---|:---|:---:|:---:|:---|:---:|:---|:---|:---|:---:|:---|:---|
|FMT-001|PaperTotalWordCount|论文总字数达标|总字数（不含参考文献 / 附录）≥3 万字（软件工程硕士体量要求）|7|CRITICAL|total_word_count|≥|30000|word|字|FMT|FormatAuditAgent|格式审计智能体|
|FMT-002|CoreChapterWordRate|核心章节字数占比达标|第 3-5 章原创研究章节字数占比≥60%（保证实质工作量）|6|CRITICAL|core_chapter_rate|≥|60%|percent|%|FMT|FormatAuditAgent|格式审计智能体|

### 4.5 执行规则：运行时配置（rules.yaml / agent_rules）
1. 执行规则用于驱动FMT检测模块的具体策略与阈值（例如错别字、术语一致性、标点混用、引用格式、RAG评语生成开关等）。
2. 执行规则的组织形式可为YAML或JSON，统一存放于：
   - 本地默认：`rules.yaml`
   - 数据库覆盖：`agent_rules`表（`rule_id`建议使用模块名，如`typo_check`、`punctuation_check`等）
3. 加载优先级：数据库规则覆盖本地同名规则；数据库缺失时回退到本地规则。

## 5. 数据库设计与操作规范（FMT必须遵守）
### 5.1 核心数据表（FMT需使用且不可改核心字段）
#### 5.1.1 论文内容表（paper_sections）
|字段|类型|说明|
|---|---|---|
|paper_id|UUID|论文全局唯一标识|
|section_name|String|章节名（abstract/intro/methodology/code/experiment等）|
|content|Text|Markdown格式的章节原始文本|

#### 5.1.2 专家知识库（expert_comments）
|字段|类型|说明|
|---|---|---|
|comment_id|String|评语唯一ID|
|metric_id|String|关联的评价指标ID（例如FMT-003）|
|text|Text|专家原始评语内容|
|embedding|Vector(768)|SBERT生成的向量，用于语义相似度检索|

#### 5.1.3 任务协同表（review_tasks）
用于记录FMT的任务状态、结果与性能数据。字段结构需保持与系统规范一致：
|字段名|类型|约束|说明|
|---|---|---|---|
|id|BigSerial|Primary Key|自增主键|
|task_id|String/UUID|Indexed|对应上传request_id|
|paper_id|UUID|Indexed|关联论文ID|
|chunk_id|String|-|切片ID|
|agent_name|String|-|Agent名称（固定为FMT对应的Agent名称）|
|agent_version|String|-|Agent版本|
|status|Enum|Default: PENDING|PENDING/RUNNING/SUCCESS/FAILED/TIMEOUT|
|score|Integer|-|从result_json冗余的分数|
|audit_level|String|-|Info/Warning/Critical|
|result_json|JSONB|-|完整出参原始数据|
|error_msg|Text|-|失败原因|
|usage_tokens|Integer|-|Token消耗|
|latency_ms|Integer|-|耗时(ms)|
|created_at|Timestamp|Default: Now|创建时间|
|updated_at|Timestamp|Default: Now|更新时间|

#### 5.1.4 反思评估输入表（agent_audit_result）
用于向反思评估组提供结构化“审计明细输入”。本表字段以系统侧实际表结构为准，其中`result_json`为必备字段：
|字段名|类型|约束|说明|
|---|---|---|---|
|result_json|JSONB|Default: {}|审计明细输入（严格按本节JSON格式输出）|

`result_json`内容格式（审计组统一输出）：
```json
{
  "agent_code": "FMT",
  "audit_results": [
    {
      "result_id": "item-001",
      "paper_id": "paper-001",
      "point": "统计学显著性检验",
      "rule_id": "FMT-006",
      "score": 3,
      "level": "Warning",
      "description": "实验三数据分布不均，未进行正态性检验。",
      "evidence_quote": "原文第4.2节提到：'我们直接采用了T检验...'",
      "location": {
        "section": "4.2",
        "line_start": 45
      },
      "suggestion": "建议补充Shapiro-Wilk检验。"
    }
  ]
}
```

约束：
1. `agent_code`为智能体编码（例如FMT/REF/EXP/LOG），用于反思评估组跨智能体聚合。
2. `audit_results`为明细数组；每个对象必须包含本节示例中的全部字段，缺省用`""`或`null`（如`line_start`）。
3. `evidence_quote`必须为原文摘录（用于幻觉过滤），禁止生成与原文无关的“伪引用”。
4. `location`用于前端跳转与幻觉过滤；本仓库实现中若无法解析章节行号，可使用页码映射（例如`section="p12"`）并尽量填充可用的`line_start`近似值。

### 5.2 数据库操作要求
1. 所有数据库操作必须使用异步驱动与异步会话，避免阻塞。
2. `paper_id/task_id/chunk_id`需与上游上传与切片拆解环节严格关联，保证一致性。
3. 向量检索仅基于pgvector执行，向量维度固定为768，不可修改。
4. FMT完成审计后必须实时写入/更新`review_tasks`表，并将审计明细同步写入`agent_audit_result.result_json`，确保状态与反思评估输入时效性。

## 6. API交互规范（FMT必须遵守）
### 6.1 论文切片上传协议（Orchestrator → FMT）
```json
{
  "request_id": "req_20231027_001",
  "metadata": {
    "paper_id": "uuid-string",
    "paper_title": "论文标题",
    "chunk_id": "chunk_seq_005"
  },
  "payload": {
    "content": "论文切片内容",
    "context_before": "前一段落摘要",
    "context_after": "后一段落开头"
  },
  "config": {
    "temperature": 0.1,
    "max_tokens": 500
  }
}
```

### 6.2 审计结果返回协议（FMT → Orchestrator）
```json
{
  "request_id": "req_20231027_001",
  "agent_info": {
    "name": "Standardization_Auditor_Agent",
    "version": "v1.1"
  },
  "result": {
    "score": 85,
    "audit_level": "Warning",
    "comment": "审计评语内容",
    "suggestion": "修改建议内容",
    "tags": ["Citation_Inconsistency", "Label_Missing"]
  },
  "usage": {
    "tokens": 120,
    "latency_ms": 1500
  }
}
```

### 6.3 接口开发要求与状态码
1. 所有接口必须为异步接口（`async def`）。
2. 入参/出参必须由Pydantic V2强制校验。
3. 输入输出JSON格式需完全匹配协议，无值字段填`null`。
4. 标准HTTP状态码：成功200、参数错误400、服务异常500。
5. 必须记录Token消耗与耗时，并写入`review_tasks.usage_tokens`与`review_tasks.latency_ms`。

## 7. FMT专属审计维度与标签（tags）
### 7.1 维度范围
FMT聚焦论文排版与格式一致性，包括但不限于：
1. 引用一致性（编号/作者年制/引用位置与标点关系）
2. 图表标号与正文引用一致性、标题位置规则（图下/表上）
3. 标题层级与序号层级一致性
4. 术语一致性、公式规范、标点符号中英文混用、列表符号统一

### 7.2 专属tags（固定枚举）
FMT输出的`result.tags`仅允许使用：
1. `Citation_Inconsistency`
2. `Label_Missing`
3. `Punctuation_Error`
4. `Hierarchy_Fault`

## 8. 核心业务逻辑规范（FMT必须遵守的三步闭环）
FMT的语义侧审计（含RAG评语生成）必须遵循三步闭环逻辑：
1. 事实提取：从论文切片抽取可核对的事实点（结构、位置、编号、约束条件等）
2. 语义检索：基于事实点向`expert_comments`向量检索，拉取最匹配的专家评语
3. 评价生成：结合原文证据、规则要求与专家话术，生成带依据的comment与suggestion

## 9. 交付与兼容性要求（FMT）
1. 版本号遵循`vX.X`格式，并确保写入`review_tasks.agent_version`可用于效果回测。
2. 交付物需包含可安装依赖清单（requirements.txt）与可运行服务入口。
3. 禁止在日志与出参中泄露敏感配置（如数据库完整连接串、密钥等）。

## 目标
让 Standardization_Auditor_Agent 在真实数据库与当前代码条件下可稳定运行，并具备可回归、可扩展、可解释的审计能力（版面+语义+RAG专家评语+任务协同）。

## 仍需补齐（数据/配置侧）
### 1) 专家评语知识库数据（影响 RAG 质量）
- 必需字段：metric_id、text（以及可选 rule_code/rule_category/severity/weight 等）。
- 建议：
  - 统一 metric_id 命名空间与规则引擎 rule_id / issue_type 的映射关系（避免“召回到评语，但无法绑定到具体指标/规则”）。
  - 用真实向量模型批量回填 embedding（当前 seed/fallback 仅用于流程验证）。

### 2) 回归 ground truth（影响评测可信度）
- 当前缺少：对每个 regression sample 的“问题清单/严重等级/证据定位（页码、bbox）”等标注结果。
- 建议：
  - 建立最小可用标注格式（JSON）并沉淀到 ground_truth_issues。
  - 先从高价值规则（如引用占位符、标题编号连续性、图表题位置）做小规模标注闭环。

### 3) LLM 侧配置与稳定性（影响语义扫描能力）
- 需要：可用的 LLM Provider Key 与对应模型参数（超时、重试、并发限制）。
- 建议：
  - 统一在 set_env.local.ps1 / 环境变量中管理，避免写入仓库。
  - 对 llm_scan 的失败路径做“降级可用”（比如仅输出结构化错误+不中断整体审计）。

## 推荐下一步落地顺序
1) 用 scripts/verify_db_connection.py 验证连接与权限 → 确保线上/本地环境一致。
2) 填充一批高质量 expert_comments（先 30~100 条，覆盖核心规则）→ 观察 RAG 召回与报告可读性。
3) 建立最小 ground truth 标注集（先 10 个样本）→ 让回归从“能跑”变成“可量化对比”。
4) 按回归结果迭代规则与模型参数（rag_eval / llm_scan / layout 阈值）。

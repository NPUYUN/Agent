## 目标
在不写入/修改数据库结构的前提下，把项目中“可在代码侧补齐”的缺口尽量补齐；同时明确哪些能力必须依赖外部要素（数据/配置/样本/标注/密钥）才能完善。

## 现状与主要缺口
- 规则来源优先级：DB(agent_rules) 覆盖 YAML(rules.yaml)；当前 agent_rules 为空，规则仅来自 rules.yaml
- RAG 专家评语：expert_comments 记录量少，且 text/metric_id 可能为空，导致检索命中后仍可能生成不出有效建议
- 回归样本：scripts/regression_samples.json 只给相对路径；paper/ 目录被 gitignore，协作与 CI 难以复现
- 真值评测：缺少“问题标注/真值集”，无法量化验证规则与 LLM 扫描效果

## 已在代码侧补齐的内容
### 1) rag_eval 配置入口补齐（无需 DB）
- 已在 rules.yaml 增加 rag_eval 配置段，提供默认开关与检索参数
- 关联能力：让 RAG 建议生成可被显式启用/禁用，并能控制 top_k 与过滤策略

涉及文件：
- [rules.yaml](file:///d:/Project/Agent/Standardization_Auditor_Agent/rules.yaml)

### 2) RAG 检索过滤策略补齐（避免“命中但为空”）
- expert_comments 检索新增过滤项（由 rag_eval 控制）：
  - require_text=true：过滤 text 为空/纯空白
  - require_metric_id=true：过滤 metric_id 为空/纯空白
  - require_active=true：仅取 active=true 或 active 为空的记录（兼容历史数据）
- 关联能力：提升建议生成稳定性与可解释性（减少“空建议/无关建议”）

涉及文件：
- [semantic_check.py](file:///d:/Project/Agent/Standardization_Auditor_Agent/core/semantic_check.py)

### 3) 回归脚本支持“缺样本时按 URL 下载”（不提交 PDF）
- regression manifest 额外支持字段（可选）：
  - url：缺文件时自动下载到指定 path
  - sha256：下载后校验，避免样本漂移
- 关联能力：让回归测试在多人/CI 环境可复现，而不需要把 PDF 提交到仓库

涉及文件：
- [run_regression.py](file:///d:/Project/Agent/Standardization_Auditor_Agent/scripts/run_regression.py)

## 仍需要外部要素才能完善的能力（代码侧无法凭空补齐）
### A) 专家评语知识库（RAG 的“料”）
- 必需：expert_comments.text（非空、可执行建议）、embedding（768维）、metric_id（与指标体系对齐）
- 直接提升：RAG 建议质量、可解释性、稳定性

### B) 回归样本与真值标注（评测闭环）
- 必需：可分享的 PDF 样本集（覆盖边界场景）+ 对应真值标注（issue_type/severity/证据位置/期望建议）
- 直接提升：可量化的回归评测、调参闭环、误报/漏报定位效率

### C) LLM Provider 的密钥与配额策略
- 必需：可用 API Key + 超时/并发/重试/预算策略
- 直接提升：llm_scan 与 rag_eval 的线上稳定性与成本可控性

## 建议落地顺序（不涉及 DB 写入）
- 先完善 regression_samples.json：为每个样本补 url/sha256，保证回归可复现
- 再补齐 expert_comments.text/metric_id（由数据侧完成），并逐步扩大知识库覆盖面
- 最后引入真值标注集，固化为自动化评测与回归门禁

# 格式审计组 (Standardization Auditor Agent) 测试说明文档

本文档旨在指导开发人员和测试人员如何验证 Standardization Auditor Agent 的功能，包括环境准备、单元测试、集成测试及手动验证流程。

## 1. 环境准备 (Prerequisites)

在运行任何测试之前，请确保已完成以下准备工作：

1.  **Python 环境**: Python 3.10.x
2.  **依赖安装**:
    ```bash
    cd Standardization_Auditor_Agent
    pip install -r requirements.txt
    ```
3.  **数据库**:
    - 启动 PostgreSQL 服务。
    - 运行自动建库脚本 (包含创建库、安装 `pgvector` 扩展、创建表结构)：
      ```bash
      python ensure_db.py
      ```
4.  **环境变量**:
    - 推荐在 Windows PowerShell 下运行 `Standardization_Auditor_Agent/set_env.local.ps1` 写入临时环境变量（该文件默认被忽略，不应提交真实 Key/密码）。
    - 或者使用系统环境变量 / `.env` 文件（本仓库默认忽略所有 `.env*` 文件）。
    - 连接内网数据库需先连学校 VPN；若账号不具备 DDL 权限，请不要用该账号运行 `ensure_db.py` / `seed_rules.py` 等可能涉及建表/建扩展的脚本。

## 2. CLI 命令行审计测试 (CLI Audit Testing)

这是目前最推荐的测试方式，可以直接对 PDF 文件生成详细的 Markdown 报告。

### 运行方式
```bash
cd Standardization_Auditor_Agent
python scripts/cli_audit.py --pdf "path/to/your/paper.pdf"
```

### 验证内容
检查项目根目录的 `paper` 文件夹是否生成以下文件（已在 .gitignore 中忽略，不会提交到仓库）：
1.  **评分报告 (`*_score_report.md`)**:
    -   包含总分（Score）和评级（Critical/Warning/Info）。
    -   包含各类型问题的数量统计。
2.  **扣分细则 (`*_deduction_details.md`)**:
    -   **视觉布局分析 (CV Layout Analysis)**: 检查图表位置、标题层级等布局问题。
    -   **语义内容分析 (LLM Semantic Analysis)**: 检查错别字、标点符号、引用格式等语义问题。
    -   确认 Qwen API 是否成功调用（如果配置了 Key，应能看到语义分析结果）。

## 3. 单元测试 (Unit Tests)

当前仓库已包含部分 unittest 用例，可使用以下命令统一运行：
```bash
python -m unittest discover -s tests
```

验证点（建议）：
- 用例全部通过（允许有少量 skip）。
- 若接口协议变更（例如响应字段调整），先更新模型与相关用例，再运行回归。

## 4. 集成/部署验证 (Integration Tests)

集成测试通过启动真实的 FastAPI 服务进程，模拟完整的 HTTP 请求流程，验证系统各模块的协同工作情况。

### 运行方式
推荐组合以下脚本（按需选择）：

1) 端到端 HTTP 调用（需要先启动服务）：
```bash
python audit_client.py
```

2) 离线回归（不依赖服务，直接跑 layout/semantic，适合批量样本快速比较）：
```bash
python scripts/run_regression.py --layout --semantic --pages 1 --scan
```

3) 数据库连通性与权限核查（可选，连接 DB 时使用）：
```bash
python scripts/verify_db_connection.py
python scripts/verify_db_connection.py --write-check
```

### 验证内容
端到端调用建议检查：
- 响应状态码 200。
- 返回 JSON 结构符合协议：包含 `request_id/agent_info/result/usage`。
- `result` 内包含 `score/audit_level/comment/suggestion/tags`（不应包含未定义字段）。

数据库脚本建议检查：
- `TCP_CONNECT` 为 OK（内网库需先连 VPN）。
- `DB_AUTH` 为 OK 后，观察 `SCHEMA_CREATE_PUBLIC` 与 `PRIV ...` 输出是否符合预期权限模型。

## 5. 手动验证 (Manual Verification)

### 5.1 服务状态页面
启动服务后，浏览器访问 `http://localhost:8000/` (或配置的端口)。
- **预期结果**: 显示 "Standardization Auditor Agent Status" 页面，包含：
  - 当前版本 (Version)
  - 运行环境 (Environment)
  - LLM 提供商 (LLM Provider)
  - 已加载规则列表 (Loaded Rules)
  - API 文档链接 (Docs)

### 5.2 规则查看接口
浏览器访问 `http://localhost:8000/rules`。
- **预期结果**: 返回 JSON 格式的规则列表，确认 `typo_check`, `heading_check` 等核心规则都在列表中。

### 5.3 数据库验证
连接 PostgreSQL 数据库，查询 `agent_rules` 表：
```sql
SELECT rule_id, content, updated_at FROM agent_rules;
```
- **预期结果**: 应包含 `rules.yaml` 中定义的核心规则（如 `typo_check`, `heading_check`, `figure_table_check` 等）。

## 6. 常见问题 (Troubleshooting)

- **TestClient Error**: 如果遇到 `TypeError: Client.__init__() got an unexpected keyword argument 'app'`，请勿使用 `TestClient` 进行复杂集成测试，优先使用 `audit_client.py` 或 `scripts/run_regression.py` 完成回归验证。
- **Database Connection Failed**: 检查 `DATABASE_URL` 环境变量是否正确，以及 PostgreSQL 服务是否运行。
- **Layout Analysis Timeout**: 如果长文档处理超时 (默认 300s)，请在环境变量中增加 `LAYOUT_ANALYSIS_TIMEOUT` 的值。
